# Tiger Compiler Architecture

This document describes the architecture of the Tiger compiler, a complete
compiler for the Tiger programming language as described in Andrew Appel's
*Modern Compiler Implementation in C/Java/ML* (the "Tiger book").

The compiler is implemented in C++17 and targets x86-64 (System V AMD64 ABI).

## Overview

The Tiger compiler translates Tiger source programs into x86-64 assembly.
It is structured as a classic multi-pass compiler with the following phases:

```
Tiger source (.tig)
       │
       ▼
  [Lexer]          tiger.lex → lex.yy.cc
       │  token stream
       ▼
  [Parser]         tiger.y → parse.tab.cc
       │  Abstract Syntax Tree (AST)
       ▼
  [Semantic Analysis]   semant.cc
       │  type-checked AST
       ▼
  [Escape Analysis]     escape.cc
       │  AST with escape flags
       ▼
  [IR Translation]      translate.cc
       │  IR tree fragments (ProcFrag / StringFrag)
       ▼
  [Canonicalization]    canon.cc
       │  linearized, trace-scheduled IR
       ▼
  [Code Generation]     codegen.cc
       │  abstract assembly (InstrList with virtual regs)
       ▼
  [Liveness Analysis]   liveness.cc / flowgraph.cc
       │  interference graph + move list
       ▼
  [Register Allocation] regalloc.cc / color.cc
       │  colored InstrList (virtual regs → physical regs)
       ▼
  [Assembly Output]     output.cc / x64frame.cc
       │
       ▼
  x86-64 assembly (.tig.s)
```

## Compilation Pipeline

The `main()` function in `src/tiger/main/main.cc` drives the pipeline:

```cpp
// 1. Parse
absyn_tree = Parse(fname);

// 2. Semantic analysis
sem::ProgSem prog_sem(std::move(absyn_tree), std::move(errormsg));
prog_sem.SemAnalyze();

// 3. Escape analysis
esc::EscFinder esc_finder(std::move(absyn_tree));
esc_finder.FindEscape();

// 4. IR translation
tr::ProgTr prog_tr(std::move(absyn_tree), std::move(errormsg));
prog_tr.Translate();

// 5–8. Canonicalization, codegen, liveness, regalloc, output
output::AssemGen assem_gen(fname);
assem_gen.GenAssem(true);  // true = perform register allocation
```

The global `frags` list (of type `frame::Frags*`) accumulates `ProcFrag` and
`StringFrag` objects during IR translation.  The assembly generator iterates
over this list to produce the final `.s` file.

## Module Descriptions

### Lexer (Lab 2)

**Files:** `src/tiger/lex/tiger.lex`, `src/tiger/lex/lex.yy.cc`

The lexer is generated by Flex from `tiger.lex`.  It tokenises the Tiger
source file and returns tokens to the parser.

Key features:
- **Nested comments**: handled with a `comment_level` counter and a `COMMENT`
  start condition.  `/* ... /* ... */ ... */` is correctly handled.
- **String literals**: handled with a `STR` start condition.  Escape sequences
  supported: `\n`, `\t`, `\"`, `\\`, `\^C` (control characters), `\ddd`
  (decimal ASCII), and `\ ... \` (whitespace-ignoring continuation).
- **Error reporting**: illegal tokens and unterminated strings/comments are
  reported via `err::ErrorMsg`.
- **Position tracking**: `char_pos` is updated by `adjust()` for accurate
  error locations.

### Parser (Lab 3)

**Files:** `src/tiger/parse/tiger.y`, `src/tiger/parse/parse.tab.cc`

The parser is generated by Bison from `tiger.y`.  It implements an LALR(1)
grammar for the Tiger language and builds the AST.

Key features:
- **Operator precedence**: arithmetic, comparison, and logical operators are
  declared with `%left`/`%right`/`%nonassoc` to resolve ambiguities.
- **Dangling else**: resolved by standard shift-reduce preference (shift wins).
- **AST construction**: each grammar rule creates the appropriate `absyn::*`
  node and returns it as the semantic value.

### Abstract Syntax Tree

**Files:** `src/tiger/absyn/absyn.h`, `src/tiger/absyn/absyn.cc`

The AST represents the syntactic structure of a Tiger program.  It has two
node families:

**Expressions (`absyn::Exp`):**
| Node | Description |
|------|-------------|
| `VarExp` | Variable reference |
| `NilExp` | `nil` literal |
| `IntExp` | Integer literal |
| `StringExp` | String literal |
| `CallExp` | Function call |
| `OpExp` | Binary operation |
| `RecordExp` | Record creation `{f1=e1, ...}` |
| `SeqExp` | Sequence `(e1; e2; ...)` |
| `AssignExp` | Assignment `v := e` |
| `IfExp` | Conditional `if-then-else` |
| `WhileExp` | While loop |
| `ForExp` | For loop |
| `BreakExp` | Break statement |
| `LetExp` | Let expression |
| `ArrayExp` | Array creation |

**Declarations (`absyn::Dec`):**
| Node | Description |
|------|-------------|
| `FunctionDec` | Function declaration (possibly mutually recursive) |
| `VarDec` | Variable declaration |
| `TypeDec` | Type declaration (possibly mutually recursive) |

**Variables (`absyn::Var`):**
| Node | Description |
|------|-------------|
| `SimpleVar` | Simple variable reference |
| `FieldVar` | Record field access `v.f` |
| `SubscriptVar` | Array subscript `v[e]` |

Each node stores a source position (`pos_`) for error reporting.

### Semantic Analysis (Lab 4)

**Files:** `src/tiger/semant/semant.h`, `src/tiger/semant/semant.cc`,
`src/tiger/semant/types.h`, `src/tiger/semant/types.cc`

Semantic analysis type-checks the AST and verifies scoping rules.

**Type system** (`types.h`):
| Type | Description |
|------|-------------|
| `IntTy` | Integer (singleton) |
| `StringTy` | String (singleton) |
| `NilTy` | Nil (singleton; compatible with any record) |
| `VoidTy` | No value (singleton; for procedures, loops, assignments) |
| `RecordTy` | Record with named fields (nominal equality) |
| `ArrayTy` | Array with element type (nominal equality) |
| `NameTy` | Named type alias (resolved via `ActualTy()`) |

**Environments:**
- `TEnv` (type environment): maps type names → `type::Ty*`
- `VEnv` (variable environment): maps names → `env::VarEntry` or `env::FunEntry`

**Checks performed:**
- Every expression has a well-defined type
- Variables and functions are declared before use
- Mutually recursive type/function groups are handled in two passes
- For-loop variables are read-only
- `break` only appears inside loops
- `nil` is only used where a record type is expected
- Array/record creation uses the correct type and field count

### Escape Analysis (Lab 5)

**Files:** `src/tiger/escape/escape.h`, `src/tiger/escape/escape.cc`

Escape analysis determines which variables must be stored in the stack frame
(as opposed to a register).  A variable *escapes* if it is accessed from a
nested function.

**Algorithm:**
1. Traverse the AST, tracking the current nesting depth.
2. For each variable declaration, record `(depth, &escape_flag)` in the
   escape environment.
3. When a variable is used, if `current_depth > declaration_depth`, set
   `escape_flag = true`.

The `escape_` flags in `absyn::VarDec` and `absyn::Field` are set by this
pass and read by the IR translator.

### IR Translation (Lab 5)

**Files:** `src/tiger/translate/translate.h`, `src/tiger/translate/translate.cc`,
`src/tiger/translate/tree.h`, `src/tiger/translate/tree.cc`

IR translation converts the type-checked, escape-annotated AST into an
intermediate representation (IR tree) suitable for code generation.

**IR tree nodes** (`tree.h`):

*Statements (no value):*
- `SeqStm` – sequence
- `LabelStm` – label definition
- `JumpStm` – unconditional jump
- `CjumpStm` – conditional jump
- `MoveStm` – assignment
- `ExpStm` – expression for side effects

*Expressions (produce a value):*
- `BinopExp` – binary arithmetic/bitwise
- `MemExp` – memory load/store
- `TempExp` – virtual register
- `EseqExp` – statement + expression
- `NameExp` – label address
- `ConstExp` – integer constant
- `CallExp` – function call

**Key translation concepts:**

*Three-way expression representation* (`tr::Exp`):
- `ExExp` – expression with a value (`tree::Exp*`)
- `NxExp` – statement with no value (`tree::Stm*`)
- `CxExp` – conditional (CJUMP with dangling true/false labels)

Each can be coerced to any other form via `UnEx()`, `UnNx()`, `UnCx()`.

*Static links:* Nested functions access enclosing variables via a chain of
frame-pointer dereferences.  The translator follows `tr::Level::parent_`
pointers and generates `MEM(MEM(...(fp)...))` chains.

*Fragments:* Each function body becomes a `frame::ProcFrag`; each string
literal becomes a `frame::StringFrag`.  Both are appended to the global
`frags` list.

### IR Canonicalization

**Files:** `src/tiger/canon/canon.h`, `src/tiger/canon/canon.cc`

Canonicalization transforms the IR tree into a form suitable for instruction
selection.  Three phases:

**Phase 1: Linearize**
- Removes all `ESEQ` nodes by hoisting their statement parts
- Wraps every `CALL` in `MOVE(TEMP(t), CALL(...))` to capture results
- Result: a flat `StmList` with no `ESEQ` nodes

**Phase 2: BasicBlocks**
- Partitions the statement list into maximal basic blocks
- Each block starts with a `LABEL` and ends with `JUMP` or `CJUMP`
- Inserts synthetic labels and jumps as needed

**Phase 3: TraceSchedule**
- Orders basic blocks into traces to minimise unconditional jumps
- Ensures every `CJUMP` is immediately followed by its false-label block
- Result: a `Traces` object (flat `StmList`) ready for instruction selection

### Code Generation (Lab 5)

**Files:** `src/tiger/codegen/codegen.h`, `src/tiger/codegen/codegen.cc`,
`src/tiger/codegen/assem.h`, `src/tiger/codegen/assem.cc`

Code generation translates the canonicalized IR into abstract x86-64 assembly
instructions using the **maximal munch** algorithm.

**Instruction types** (`assem.h`):
- `OperInstr` – general operation (arithmetic, memory, call, jump)
- `LabelInstr` – label definition
- `MoveInstr` – register-to-register move (special: candidate for coalescing)

**Assembly string notation:**
- `` `s0, `s1, ... `` – source (Use) temporaries
- `` `d0, `d1, ... `` – destination (Def) temporaries

**Key patterns** (x86-64 AT&T syntax):
```
MOVE(MEM(BINOP(+, r, CONST(k))), src)  →  movq `s0, k(`s1)
MOVE(dst, MEM(BINOP(+, r, CONST(k))))  →  movq k(`s0), `d0
BINOP(+, r, CONST(k))                  →  addq $k, `d0
BINOP(*, r1, r2)                        →  imulq `s2  (saves/restores %rax, %rdx)
BINOP(/, r1, r2)                        →  cqto; idivq `s2
CALL(NAME(f), args)                     →  callq f
NAME(l)                                 →  leaq l(%rip), `d0
BINOP(+, r, NAME(fs))                   →  leaq fs(`s0), `d0  (frame pointer)
```

### Liveness Analysis (Lab 6)

**Files:** `src/tiger/liveness/liveness.h`, `src/tiger/liveness/liveness.cc`,
`src/tiger/liveness/flowgraph.h`, `src/tiger/liveness/flowgraph.cc`

**Control flow graph** (`flowgraph.h`):
- One node per instruction
- Edges for sequential execution and jumps
- Built by `FlowGraphFactory::AssemFlowGraph()`

**Liveness analysis** (`liveness.h`):
- Backward dataflow analysis
- Equations: `live-in[n] = use[n] ∪ (live-out[n] − def[n])`
             `live-out[n] = ∪ { live-in[s] | s ∈ succ[n] }`
- Iterates until fixed point

**Interference graph** (`IGraph`):
- One node per temporary (precolored machine registers first)
- Edge (u, v) if u and v are simultaneously live
- Move instructions: source and destination do NOT interfere (move-related)
- Precolored nodes have infinite degree (never simplified/spilled)

### Register Allocation (Lab 6)

**Files:** `src/tiger/regalloc/regalloc.h`, `src/tiger/regalloc/regalloc.cc`,
`src/tiger/regalloc/color.h`, `src/tiger/regalloc/color.cc`

Implements **iterated register coalescing** (Appel & George 1996).

**Algorithm steps:**
1. **Build**: construct interference graph via liveness analysis
2. **MakeWorklist**: classify nodes into simplify/freeze/spill worklists
3. **Simplify**: remove low-degree, non-move-related nodes
4. **Coalesce**: merge move-related pairs (George's or Briggs' test)
5. **Freeze**: give up coalescing a low-degree move-related node
6. **SelectSpill**: choose a high-degree node to potentially spill
7. **AssignColors**: assign colors (registers) to nodes on the select stack
8. **RewriteProgram**: if spills occurred, insert load/store code and restart

**Coalescing safety tests:**
- *George's test*: for every neighbor t of v, t interferes with u OR degree(t) < K
- *Briggs' test*: |{t ∈ adj(u) ∪ adj(v) | degree(t) ≥ K}| < K

**Spill rewriting:**
- Allocates a new frame slot for the spilled temp
- Replaces each use with: `movq slot(%rsp), t_new` before the instruction
- Replaces each def with: `movq t_new, slot(%rsp)` after the instruction

### Assembly Output

**Files:** `src/tiger/output/output.h`, `src/tiger/output/output.cc`,
`src/tiger/frame/x64frame.h`, `src/tiger/frame/x64frame.cc`

`output::AssemGen::GenAssem()` iterates over all fragments:

For each `ProcFrag`:
1. Canonicalize the IR tree (Linearize → BasicBlocks → TraceSchedule)
2. Generate abstract assembly (CodeGen)
3. Optionally perform register allocation (RegAllocator)
4. Call `ProcEntryExit3` to generate prologue/epilogue
5. Write `.globl`, `.type`, prologue, body, epilogue, `.size` to the output file

For each `StringFrag`:
- Write label, `.long <length>`, `.string "<content>"` to `.rodata`

**ProcEntryExit functions:**
- `ProcEntryExit1(frame, stm)`: prepend view-shift + callee-save, append callee-restore
- `ProcEntryExit2(il)`: append return-sink pseudo-instruction (keeps callee-saves live)
- `ProcEntryExit3(frame, il)`: generate prologue (`subq $fs, %rsp`) and epilogue (`addq $fs, %rsp; retq`)

## Supporting Modules

### Symbol Table

**Files:** `src/tiger/symbol/symbol.h`, `src/tiger/symbol/symbol.cc`

`sym::Symbol` is an interned string: each unique string has exactly one
`Symbol` object, so pointer equality can be used for fast name comparison.
Used for variable names, type names, function names, and assembly labels.

### Temporaries and Labels

**Files:** `src/tiger/frame/temp.h`, `src/tiger/frame/temp.cc`

- `temp::Temp`: a virtual register, identified by a unique integer
- `temp::Label`: an assembly label (alias for `sym::Symbol`)
- `temp::Map`: maps `Temp*` to register-name strings; supports layering
- `temp::TempList`: ordered list of temps with set operations (Union, Diff, Contain)

### Graph Utilities

**Files:** `src/tiger/util/graph.h`, `src/tiger/util/table.h`

`graph::Graph<T>` is a generic directed graph with typed nodes.
Used for:
- `fg::FGraph` (control flow graph): `Graph<assem::Instr>`
- `live::IGraph` (interference graph): `Graph<temp::Temp>` (extended)

`tab::Table<K, V>` is a hash table mapping `K*` to `V*`.
Used for environments (TEnv, VEnv), temp maps, and label maps.

### Runtime Library

**File:** `src/tiger/runtime/runtime.c`

Provides the Tiger runtime functions called by compiled programs:
- `print(s)`, `flush()`, `getchar()` – I/O
- `ord(s)`, `chr(i)` – character conversion
- `size(s)`, `substring(s,f,n)`, `concat(s1,s2)` – string operations
- `not(i)` – logical negation
- `exit(i)` – program termination
- `alloc_record(size)` – allocate a record on the heap
- `init_array(size, init)` – allocate and initialise an array
- `string_equal(s1, s2)` – string comparison

## Data Flow Between Phases

```
absyn::AbsynTree
    │ (sem::ProgSem::SemAnalyze)
    │ type-checks, annotates with type::Ty*
    ▼
absyn::AbsynTree  (same tree, now type-annotated)
    │ (esc::EscFinder::FindEscape)
    │ sets VarDec::escape_ and Field::escape_
    ▼
absyn::AbsynTree  (same tree, now escape-annotated)
    │ (tr::ProgTr::Translate)
    │ produces frame::ProcFrag and frame::StringFrag objects
    ▼
frame::Frags  (global list of fragments)
    │ (output::AssemGen::GenAssem, per ProcFrag)
    │
    ├─ canon::Canon::Linearize()
    │      → tree::StmList (no ESEQ, CALLs wrapped)
    ├─ canon::Canon::BasicBlocks()
    │      → canon::StmListList (basic blocks)
    ├─ canon::Canon::TraceSchedule()
    │      → canon::Traces (ordered for instruction selection)
    │
    ├─ cg::CodeGen::Codegen()
    │      → cg::AssemInstr (InstrList with virtual regs)
    │
    ├─ fg::FlowGraphFactory::AssemFlowGraph()
    │      → fg::FGraph (control flow graph)
    ├─ live::LiveGraphFactory::BuildIGraph() + Liveness()
    │      → live::LiveGraph (interference graph + move list)
    │
    ├─ col::Color()
    │      → col::Result (coloring map + spilled nodes)
    ├─ ra::RegAllocator::RegAlloc()  [iterates until no spills]
    │      → ra::Result (final InstrList + coloring map)
    │
    └─ frame::ProcEntryExit3()
           → assem::Proc (prologue + body + epilogue strings)
               → written to .tig.s
```

## Key Design Decisions

### 1. Machine-independent frame interface
`frame::Frame` and `frame::RegManager` are abstract base classes.  The
concrete x86-64 implementation (`X64Frame`, `X64RegManager`) is isolated in
`x64frame.h/cc`.  Adding a new target architecture requires only a new
concrete subclass.

### 2. Three-way IR expression representation
`tr::Exp` has three variants (ExExp, NxExp, CxExp) following Appel Chapter 7.
This avoids generating unnecessary comparisons for boolean expressions and
allows efficient translation of `if-then-else` and loop conditions.

### 3. Iterated register coalescing
The register allocator implements the full Appel & George algorithm including
coalescing (merging move-related temps) and spill rewriting.  This produces
high-quality register assignments with few unnecessary moves.

### 4. Frame pointer via leaq
Instead of using `%rbp` as a traditional frame pointer, this compiler
computes the frame pointer as `SP + framesize` using a `leaq` instruction.
This frees `%rbp` for use as a general-purpose callee-saved register.

### 5. Static links for nested functions
Tiger supports nested function definitions.  The compiler passes a static
link (pointer to the enclosing frame) as the first argument to every function
call.  The translator generates chains of `MEM` dereferences to access
variables in enclosing scopes.

### 6. Ownership transfer pattern
Each compilation phase takes ownership of its input (via `std::unique_ptr`
and `std::move`) and returns ownership of its output.  This makes the
pipeline explicit and prevents use-after-free bugs.

## Directory Structure

```
src/tiger/
├── absyn/          Abstract syntax tree (AST) nodes
├── canon/          IR tree canonicalization (linearize, basic blocks, traces)
├── codegen/        Instruction selection (maximal munch) + assembly types
├── env/            Type and variable environments (built on symbol table)
├── errormsg/       Error reporting with source positions
├── escape/         Escape analysis (which variables must be in the frame)
├── frame/          Activation records, register manager, temps, labels
├── lex/            Lexer (Flex source + generated code)
├── liveness/       Control flow graph + liveness analysis + interference graph
├── main/           Compiler driver and test drivers
├── output/         Assembly output coordination
├── parse/          Parser (Bison source + generated code)
├── regalloc/       Register allocation (iterated register coalescing + coloring)
├── runtime/        Tiger runtime library (C)
├── semant/         Semantic analysis (type checking + scoping)
├── symbol/         Interned symbol table
├── translate/      IR tree nodes + AST-to-IR translation
└── util/           Generic graph and hash table utilities
```
